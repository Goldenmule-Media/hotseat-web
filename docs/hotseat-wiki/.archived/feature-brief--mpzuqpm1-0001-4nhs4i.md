# Feature: wiki-ui interactive FSM transitions

**Status:** shipped

## Summary
The model-inspection view (the shipped "wiki-ui model inspection" feature) makes a page's governance legible: it draws the page type's status FSM as a directed graph, highlights the page's current state, and colors each outgoing transition available, blocked-with-reason, or inert. But the view is read-only. To actually move a page along its lifecycle you still drop out of the UI and call describeMutations + mutatePage by hand (over MCP), or edit the page some other way — the graph shows you the door but won't open it.

This feature makes the graph actionable. Click a transition that leaves the current state and a form opens, generated from that command's argument schema; fill in the structured data and a single button runs the mutation over that edge. Because wiki-ui already live-tails the workspace, the resulting commit flows back through the existing subscription and the graph repaints at the new state — the current-state marker moves and the available set re-colors with no manual reload.

The form is schema-driven, not hand-written: describeMutations already returns each command's argsSchema (a JSON Schema) alongside its availability, and a transition edge's event IS the command name, so the clicked edge maps to exactly one descriptor whose schema the form renders. No per-command UI, nothing hardcoded per page type. The engine stays the single validator — the form coerces inputs and submits, then surfaces the engine's ValidationError verbatim rather than re-implementing the command's rules. A blocked edge opens the same form read-only, showing the unmet precondition.

A caveat worth stating plainly: in the current models every page-level transition takes empty args, so today the form is a confirm-and-run step ("Run beginPlanning →") rather than a data-entry form. That is still useful — it advances a page's lifecycle from the UI and makes precondition gating actionable — and the mechanism renders real input fields the instant any transition declares arguments. Keep the two altitudes distinct, as the model view does: the graph and each command's arg schema belong to the page TYPE; availability and the act of running a transition belong to the page INSTANCE.

## Components affected
- wiki-ui FsmGraph (components/FsmGraph.tsx) — make edges leaving the current state clickable (React Flow onEdgeClick), stash {event, from, to, cls} on each edge, and open the transition form for the clicked command
- wiki-ui pure schema→form transform (lib/schema-form.ts) — maps a command's argsSchema (JSON Schema) to ordered field specs + coerces raw inputs to typed args; unit-tested, no React (mirrors the buildFsmGraph split)
- wiki-ui TransitionForm + Modal (new components) — a schema-driven form (confirm-only when the schema has no fields, read-only + reason when the edge is blocked) inside a lightweight dialog overlay; surfaces the engine's validation errors
- wiki-ui live store (lib/live.tsx) — a write path (usePageMutator) over the existing WorkspaceSession IWorkspaceHandle.mutate, with pending/error state; the existing live tail does the post-commit repaint (no polling)
- wiki engine (wiki) — NO change required: IMutationDescriptor.argsSchema + IWorkspaceHandle.mutate / IPageView.mutate already exist, and describeMutations already returns the args schema, availability, and the unmet reason. The feature reuses this surface rather than extending it

## Design constraints
1. The form is generated from the command's machine-readable argument schema — the argsSchema describeMutations already returns — never hand-authored per command and never hardcoding any page type's commands or fields. An unrecognized schema shape degrades to a raw-JSON input rather than failing to render.
2. The engine is the only validator. The UI may hint required/type from the JSON Schema, but must not re-implement the command's Zod rules; on submit it calls mutate and surfaces the engine's ValidationError verbatim. A second client-side validator would drift from the authoritative one.
3. Reuse the existing in-browser engine and live tail: perform the write through the same WorkspaceSession IWorkspaceHandle the reads already hold, and let the existing subscribe path repaint the graph on the resulting commit. No polling, no second engine, and no optimistic local mutation — the committed event is the single source of the post-transition view.
4. Only transitions leaving the current state are interactive: an available edge opens a runnable form, a blocked edge opens it read-only with the unmet reason, and an inert edge is not clickable. Availability and the unmet reason come from the same describeMutations overlay that already colors the graph, so the form can never offer a transition the engine would reject on status or precondition grounds.
5. Respect the type/instance split the model view established: the FSM graph and each command's argument schema are properties of the page TYPE; availability and the act of running a transition are properties of this page INSTANCE. The feature adds the instance-level action without hardcoding any type-level knowledge in the UI.
6. Keep the non-trivial logic in a pure, unit-tested module (schema→fields and value coercion); the React parts (dialog, form, edge-click wiring) stay thin. This matches wiki-ui's existing buildFsmGraph / FsmGraph split and keeps the testable core free of React and React Flow.

## Resolved questions
1. **Which edges are interactive, and what happens when you click a blocked or inert edge? The model view already classifies each outgoing edge as available (green), blocked-with-reason (amber), or inert (grey, not leaving the current state). Does the new click action attach to all three, only available, or available + blocked?** — _Only edges that leave the page's current state are interactive. An available (green) edge opens a runnable form; a blocked (amber) edge opens the same form read-only, showing the unmet precondition ("here's why you can't ship yet"); an inert (grey) edge — one that doesn't originate at the current state — is not clickable. This maps the new action exactly onto the three classes buildFsmGraph already computes from the describeMutations overlay, so no new code has to decide legality._
2. **Where does the form live — a modal dialog over the graph, a docked side panel, or an inline drawer below the canvas? This affects how the type-level graph and the instance-level action sit together visually, and how much layout work v1 takes on.** — _A modal dialog overlaid on the graph (backdrop, Escape / click-outside to dismiss, focus on the first field). It focuses attention on the single decision while the current-state highlight behind it preserves context, and it adds no layout reflow to the existing React Flow canvas. A docked side panel or inline drawer is a reasonable later refinement but buys nothing for v1; the dialog is the simplest accessible pattern that leaves the graph untouched._
3. **How is the form built — hand-written per command, or generated from the command's argument schema? describeMutations already returns an argsSchema (JSON Schema) per command. Do we drive the form fields from that schema generically, or author bespoke inputs for each transition?** — _Generated from the command's argsSchema. describeMutations already returns a JSON Schema per command, and a transition edge's event IS the command name, so the clicked edge resolves to exactly one descriptor. A pure schemaToFields(jsonSchema) maps the object schema (properties + required — the shape zod-to-json-schema emits) to typed field specs (string / number / boolean / enum / array), which the renderer draws; an unrecognized shape falls back to a raw-JSON field so the form never fails to render. No per-command UI and nothing hardcoded per type — the form is correct for any future transition that gains arguments, which is also what constraint #1 requires._
4. **Validation — replicate the command's rules client-side for instant feedback, or let the engine be the sole validator? The command args are Zod-validated at commit. Do we mirror those rules in the UI (risking drift), or submit and surface the engine's error?** — _The engine is the sole validator. The form does only light, schema-derived hinting — mark required fields, coerce text to number/boolean, drop empty optionals — then submits; the engine's parse throws a ValidationError whose message the form shows inline next to the run button. Re-implementing each command's Zod schema in the UI would duplicate logic that can silently drift from the authoritative validator, so the UI never runs a second validator; it formats and forwards the engine's verdict._
5. **How is the mutation actually performed from the browser, and how does the graph update afterward? wiki-ui runs the engine in the browser and tails the stream read-only today. Do we add a server endpoint for writes, or write through the in-browser engine handle — and what drives the post-transition repaint?** — _Through the engine handle the read side already owns: WorkspaceSession.handle() → IWorkspaceHandle.mutate(pageId, command, args). The Durable Stream host already permits cross-origin writes (it sets access-control-allow-origin '*' and allows POST/PUT/HEAD/OPTIONS), so the in-browser engine can append directly — no new server endpoint, matching the existing read-only-direct-tail architecture. On a successful commit the form closes; the existing subscribe tail receives the event, bumps lastEventAt, and the page's status + describeMutations re-read, so FsmGraph repaints at the new state and re-colors the available set. Read-your-writes comes for free, with no polling and no optimistic local state._
6. **Today every page-level transition in the loaded models takes empty args — is a generated args form still worth building, and what is the empty-args experience? If clicking a transition yields no fields, is the feature still valuable, and how should an empty schema render?** — _Yes — the schema-driven mechanism is the deliverable, and hardcoding per-command forms is explicitly disallowed (constraint #1), so we build the generic form regardless. For an empty schema it renders zero fields and becomes a confirm-and-run step ("Run beginPlanning →"). That is genuinely useful today: it advances a page's lifecycle from the UI and makes precondition gating actionable — a blocked transition shows its reason and an available one runs on click — instead of dropping to a hand-written mutatePage call. So empty-args is a clean confirmation dialog, not a degenerate empty form, and it grows input fields automatically the instant any transition declares arguments._
7. **Scope — page-status transitions only, or element-level transitions and free content commands too? Several element types carry their own FSMs and the richest-argument commands (addComponent, answerQuestion, setSummary) are content edits, not page-status edges. Is in-graph running of those in scope for v1?** — _Page-status FSM transitions only — the edges this graph actually draws — matching the v1 model-inspection scope. Element-level FSMs (a question open→resolved, a step todo→done, a case planned→passed) and free content commands (addComponent, answerQuestion, setSummary) carry the richest arguments, but they are not edges in the page-status graph; running them in-graph becomes meaningful only once element FSMs are themselves visualized, which is the established follow-up to model inspection. Holding v1 to page transitions keeps this to one focused increment that reuses the existing graph wholesale._
