# Feature: Full-text search — an engine-bundled search index for client and server

**Status:** shipped

## Summary
Today the only way to find a page is the `search` tool, and it matches a case-insensitive substring against page **titles** only (`wiki-mcp/src/mcp/tools.ts`) — it never looks inside a page. But a page's substance lives in its body: prose, code blocks, constraints, questions, decisions. Title-search misses almost all of it — you cannot find the page that discusses "optimistic concurrency" unless those words happen to be in its title. As the wiki grows into the system's source of truth (architecture, feature specs, and soon ADRs), "I know we wrote this down somewhere" needs a real answer.

This feature adds full-text search over page **content**: a derived search index that ingests each page's deterministic Markdown render, tokenizes and ranks it, and answers ranked queries with the pages that match — each with a highlighted snippet of surrounding context. Because the engine already renders every page deterministically to Markdown from folded state, the text to index is already there and already canonical; the index is one more read-side projection over it.

The twist is in where it lives and what it depends on. The index is bundled in the engine (`wiki/`) so a single implementation serves both consumers that fold state in-process — wiki-ui in the browser (which subscribes to the stream and folds locally, with no SQL read model) and the wiki-mcp host on the server. The engine takes a hard dependency on **Kysely alone** and never bundles a database driver: the container injects the DB. wiki-server passes pg or PGlite; the browser passes PGlite writing to IndexedDB; a CLI passes whatever it likes — all Postgres-compatible, so one Kysely-based index runs everywhere. This is the exact injection seam wiki-mcp already uses for its SQL read model, pulled one layer down into the engine. It stays schema-agnostic — it indexes the rendered Markdown, not any concrete page type — so the engine's transport-free, schema-agnostic contract holds. It would be the engine's first real content projection, maintained off the same in-process fold that today only bumps `InMemoryReadModel`'s applied version.

Architecturally this is the projection pattern the host already runs (ADR-M3: a projection is fold + serialize), pulled down to run anywhere the engine runs, and it realizes a line `wiki-mcp`'s own DESIGN §14 already carries as future work — "a search index … behind the same `IReadModel`." Full-text itself is Postgres FTS (`to_tsvector` / `to_tsquery` / `ts_rank`), which both pg and PGlite provide, so ranking and `ts_headline` snippets come from the database rather than hand-rolled code. It is exposed through MCP by upgrading the `search` tool from title-substring to true full-text — same cross-workspace fan-out, same consistency-token gating for read-your-writes — so an agent's "find where we decided X" returns the actual page.

## Components affected
- wiki (engine) — a new schema-agnostic search-index read model built on Kysely: a compact FTS schema (one search-document row per page, holding the rendered Markdown and a `tsvector` column) plus index / remove / query operations, maintained by an in-process projection off the fold (the same hook that drives `InMemoryReadModel.notifyApplied`). It indexes each page's deterministic Markdown render. The engine depends only on Kysely; the DB is injected. This is the engine's first content-bearing read projection.
- wiki (engine) render — the indexed document is exactly `renderPage`'s output (the same Markdown wiki-ui and the disk projection use); no separate text-extractor and no per-field walk. Postgres `to_tsvector` tokenizes that document and `ts_headline` produces the highlighted snippet; an optional `setweight` over the title's H1 line is a cheap later refinement if title hits should outrank body hits.
- wiki-mcp — construct and inject the database (PGlite locally / pg in prod), reusing the very Kysely instance that already backs the SQL read model so the FTS table lives beside the read-model tables; feed the index from the existing projection tailer so it stays current with the durable stream and rebuilds on boot; and back the `search` tool with it.
- wiki-mcp MCP surface — upgrade the `search` tool from title-substring to full-text over content: ranked hits with snippets, keeping the cross-workspace fan-out and consistency-token gating already in `tools.ts`.
- wiki-ui (browser) — gains content search for free by querying the same engine index it already folds into: a search box over the locally-folded workspace, no server round-trip required.
- DESIGN.md (wiki + wiki-mcp) — document the new engine search-index seam and its determinism/rebuild contract; retire the wiki-mcp §14 "full-text search" future-work line.

## Design constraints
1. Schema-agnostic: the index ingests the page's deterministic Markdown render and never concrete page-type knowledge, so the engine stays schema-agnostic and the index works for any model bundle (architecture, feature, toc, adr, …).
2. Engine depends only on Kysely; the container injects the database. The engine ships the search-index schema and queries against a Kysely handle and never bundles a driver — wiki-server passes pg or PGlite, the browser passes PGlite (persisting to IndexedDB), a CLI passes its own. All are Postgres-compatible, so one Kysely-based implementation runs in every container. This is the same injection seam wiki-mcp already uses for its SQL read model.
3. Derived and rebuildable: the index is a projection, never a source of truth — it can always be reconstructed by folding the stream, so a cold start (or a fresh browser tab) rebuilds it. Persistence, if any, is an optimization, not a correctness requirement.
4. Eventually-consistent, never on the write path: indexing runs behind the commit on the read side (CQRS); a write is never slowed by indexing work.
5. Read-your-writes: a search is a token-gated read like every other — it waits on the high-water token of each workspace the session has written (the fan-out already in `search`), so results reflect the caller's own latest commits.
6. Deterministic, dialect-pinned text search: tokenization, stemming, and ranking come from the injected database's full-text engine (Postgres `tsvector` / `tsquery` / `ts_rank`), pinned to a single text-search configuration so that pg and PGlite return identical hits and ranking for the same query and state. The configuration (and any custom dictionary) is versioned with the index.
7. Incremental: a commit re-indexes only the pages whose version changed, not the whole workspace; archived/removed pages drop out of (or are flagged in) the index following the same visibility flag the tree already uses.

## Resolved questions
1. **The "db" / index technology: a pure-TypeScript inverted index bundled in the engine (zero dependencies, loads identically in browser and Node, fully under our control — but we own ranking and scale ourselves), or an embedded SQL FTS — SQLite FTS5 or PGlite's Postgres `tsvector` (mature BM25 ranking and proven scale, but pulls an environment-specific WASM/native driver into the previously dependency-light, transport-free engine, with different drivers in browser vs Node)? Constraint #2 leans toward the pure-TS index; the trade is ranking quality and scale vs. keeping the engine light and uniform.** — _Neither a pure-TS index nor a database baked into the engine. The engine depends only on **Kysely** and the container injects the DB — pg or PGlite on the server, PGlite (persisting to IndexedDB) in the browser, whatever a CLI prefers — the exact seam wiki-mcp already uses for its SQL read model. Full-text is Postgres FTS (`to_tsvector` / `to_tsquery` / `ts_rank`), which both pg and PGlite provide, so one Kysely-based implementation serves every container. It also hands us mature ranking and `ts_headline` snippets from the database, retiring the "we'd own ranking and scale ourselves" worry. The only constraint on the injected DB is that it be Postgres-compatible (so the FTS SQL is uniform); supporting a non-Postgres dialect later would mean abstracting the FTS query behind a small strategy, not changing the seam._
2. **One backend or two: does the server reuse this engine index, or keep full-text in wiki-mcp's existing Postgres read model (`tsvector` over the page text it already stores) and let the engine index serve only the browser — i.e. one shared implementation, or two backends behind a single query contract? "Bundled in the engine because we want it on both" leans to one shared implementation; the counter is that the server already has Postgres and could get stronger FTS almost for free.** — _One implementation, not two. The engine's Kysely-based FTS is the single implementation and the single query contract; the container only chooses the DB. On the server that DB can be the *same* Kysely/Postgres instance that already backs the SQL read model — the FTS table simply lives beside the read-model tables — so it isn't "two backends" so much as one engine index targeting whatever Postgres-compatible database it is handed. The browser hands it a PGlite instance; the server hands it pg or PGlite; the index code is identical._
3. **What text is indexed: the page's deterministic Markdown render (simple — exactly what a human reads), or structured field-extraction (index prose / scalars / code / block-inlines per section, with field-level weighting, title boost, and code handled specially)? Field-extraction gives richer relevance and section-level hits at the cost of more index machinery.** — _Index the page's deterministic Markdown render — the exact output of `renderPage`, one search document per page. No separate text-extractor and no per-field walk: the render is already canonical, already what wiki-ui and the disk projection show, and already includes prose, code fences (so identifier search works alongside the symbol index), and the title as its H1. Postgres `to_tsvector` tokenizes that document; `ts_headline` returns the highlighted snippet. Trade-off accepted: the render also carries section headings and the odd `_None._` / `## Child pages` boilerplate, so we strip obvious boilerplate if it proves noisy — but one-document-per-render is the point, and we keep it simple._
4. **Granularity and result shape: page-level hits only, or section/field-level hits with snippets and highlighted match spans? And how much ranking — title-boosted term frequency, or BM25-style scoring? In short, what should a `search` result return to an agent (ids + titles, vs. ranked snippets it can read without a second fetch)?** — _Page-level hits, each with a `ts_headline` snippet (surrounding context, match terms highlighted) and a rank — the same shape everywhere; the difference is what each consumer does with it. In **wiki-ui**, a hit shows its snippet and clicking it opens the page and scrolls to the match: the UI re-finds the matched text (or its nearest preceding heading) in the same render it already displays, so "jump to that section" is a client affordance, not something the index models. Through **MCP**, a hit is the page (id + title) plus the snippet and rank; there is no reliable way to map a character offset in the flat render back to a structured section/field id, so MCP stays page-grained. Section-grained MCP hits would mean indexing per section (splitting the render by heading, or reviving per-field extraction) — a later, drop-in change to the indexer that does not touch the injection seam. Not now._
5. **Tool surface: evolve the existing `search` tool in place (title-substring → full-text, same name and fan-out), or add a separate `searchText` / `fullTextSearch` tool and keep `search` as the fast title-only finder? Evolving keeps one obvious entry point; adding preserves a cheap title lookup alongside the heavier content query.** — _Evolve `search` in place — no new `searchText`. Same name, same signature (`{ query, workspaceId? }`), same cross-workspace fan-out and consistency-token gating; it stops being title-substring and becomes full-text over the rendered document, returning ranked hits with snippets. Nothing is lost by evolving rather than adding: the title is the H1 of every render, so every title match is still a match — `search` strictly gains reach. One obvious entry point, and existing callers keep working._
6. **Persistence: keep the index purely in memory and rebuild from the fold on every boot/tab (simplest, always correct), or persist it (server: alongside the SQL read model; browser: IndexedDB/OPFS) and update incrementally from the applied cursor? Treat persistence as a deferred optimization until rebuild cost is shown to matter, or design for it from the start?** — _Persistence is the injected database's job, not a separate concern the engine owns: PGlite→IndexedDB in the browser, pg or PGlite on the server. The index stays derived and rebuildable (constraint #3) — folding the stream reconstructs it after a wipe or an index-schema-version bump — but in steady state it is durable wherever the container's DB persists, updated incrementally from the applied cursor rather than rebuilt on every boot or tab. So: designed for persistence from the start, because the injected DB gives it for free, with full-rebuild always available as the recovery path._
