# Feature: ADR page type — decisions as first-class wiki pages

**Status:** shipped

## Summary
The decisions that govern this system live as ADR appendices pinned to the bottom of five separate DESIGN.md files — ADR-001…010 in `wiki`, ADR-M1…M7 in `wiki-mcp`, ADR-S1…S3 in `wiki-server`, ADR-W1 in `wiki-models`, and ADR-C1…C5 in a `wiki-cli` package that does not yet exist. They are flat prose: no status, no lifecycle, no link between a decision and the one that revises it, and identity is per-file rather than global — `wiki-mcp` and `wiki-models` already ship two *different* records both labelled "ADR-M7", a collision a single namespace would have made impossible. There is no way to mark a decision superseded, to ask "which decisions touch the read model", or to follow a decision's consequences.

This feature adds an `adr` bundle to `wiki-models` defining a `decision-record` page type: a typed page with Context / Decision / Consequences, a status FSM (`proposed → accepted`, then `superseded` or `deprecated`; `proposed → rejected`), a date, deciders, and a `supersededBy` reference to the record that replaces it. Design decisions become first-class wiki pages — gathered in one ADR workspace, globally identified, searchable, FSM-governed, and cross-linked — instead of inert text siloed per package.

This is the move already made for the codebase's own architecture (the `architecture` bundle, with its current/stale FSM and cross-page dependency refs) and for its feature specs (this very TOC). The engine and host stay schema-agnostic; the ADR type is one more bundle loaded at runtime. Crucially, supersession is enforced by the engine's link integrity — a record can only enter `superseded` once it names a *live* successor — so the decision graph cannot silently rot the way prose appendices do.

Because the engine renders deterministically to Markdown, the ADR workspace can be projected back to `docs/adr/*.md` so git keeps a diffable snapshot — but that disk projection is a separate feature (see the linked "Markdown projection to disk" spec). This spec covers the page type itself, the migration of the existing ~27 ADRs into the wiki, and the in-wiki experience. The point is reflexive: the decisions that govern the wiki become governed objects inside it.

## Components affected
- wiki-models — new `adr` bundle (`src/adr/adr.ts` + `index.ts`): the `decision-record` page type (sections, status FSM, fields, `supersededBy` ref, render config) and its default-exported page-type array, the contract the ModelRegistry loader expects.
- Migration (one-off, `wiki-models/scripts/`) — a re-runnable script that parses each package's DESIGN.md ADR appendix into `decision-record` pages in a fresh "ADRs" workspace, preserving the legacy label, date, and scope, and wiring `supersededBy` where the prose already says one decision replaces another.
- wiki-models/DESIGN.md — document the new bundle; author a meta-ADR ("design decisions live in the wiki") as the first record.
- wiki engine (`wiki`) — expected to need NO change: the type must be expressible with existing field-kinds (prose / blocks / scalar / list / ref) and the existing FSM + `Precondition` machinery. Any gap (e.g. a ref to an arbitrary same-workspace page) is surfaced as an engine question, not worked around in the model.
- docs/adr/ — the rendered snapshot target, produced by the Markdown-to-disk projection (linked spec), not written by this feature directly.
- wiki engine (`wiki`) — ONE small, generic change: teach `kindFor` (`command-bus.ts`) the `ref` case so the declarative `set:` sugar builds a page-ref from a string id (and passes a structured ref through). Schema-agnostic — it completes `ref` support in `set:` for ALL models, with zero ADR knowledge in the engine. Everything else the type needs (`setField` with a ref value; ingestion ref-integrity) already exists and is proven by `architecture`'s page-ref field.

## Design constraints
1. Schema-agnostic boundary holds: the ADR type ships only in `wiki-models` and loads at runtime; no concrete ADR knowledge enters `wiki`, `wiki-mcp`, or `wiki-server`.
2. Determinism: equal ADR state must render byte-identical Markdown. The decision date is a stored field (never `new Date()`), and the ordering of consequences and relations is explicit — so the eventual disk snapshot produces no spurious diffs.
3. Global identity: every ADR is one page in one ADR workspace with a stable global id. The legacy per-file label (e.g. "wiki/ADR-004", "wiki-mcp/ADR-M7") is preserved as a field for traceability, but is no longer the identity — which is what dissolves the ADR-M7 collision.
4. Supersession is a real, integrity-checked edge, not prose: entering `superseded` requires a `supersededBy` ref to an existing `decision-record` (the engine rejects a dangling target), and the render shows both directions — "Supersedes …" and "Superseded by …".
5. Faithful to the established ADR form — Context → Decision → Consequences, plus status, date, and deciders — so migrated records read like the originals and the type is reusable for any project, not hardwired to this repo.
6. Reuse, don't reinvent: model the lifecycle with the existing status-FSM + `Precondition` machinery and the existing field-kinds. If the type cannot be expressed without an engine change, that is a finding to surface (a gap in the authoring surface), not a reason to special-case the engine.

## Resolved questions
1. **Type name and id prefix: `adr` (terse prefix `adr:<id>`, but jargon), `decision-record` (self-describing, longer prefix, consistent with `feature-brief`/`implementation-plan`), or `decision`? The id prefix is the type tag, so it shows up in every page id, tree row, and cross-page link.** — _`decision-record` (id prefix `decision-record:<id>`), with the human label "ADR". It is self-describing in the tree and in links and matches the descriptive style of the existing `feature-*` types; "ADR" survives as the display label and as the preserved legacy-id field, so nothing reads as jargon and nothing is lost._
2. **Status FSM: what states and transitions model a decision's life? A minimal `proposed → accepted` plus terminal revisions, or something richer? This graph is what the model-inspection view will draw and what gates legal mutations.** — _Initial `proposed`; `proposed --accept--> accepted`, `proposed --reject--> rejected`, `accepted --supersede--> superseded`, `accepted --deprecate--> deprecated`. `rejected`, `superseded`, and `deprecated` are terminal. `supersede` is precondition-gated on naming a successor (Q4). This mirrors how ADRs actually move — most live their life in `accepted`; a few are later replaced or retired — and keeps the graph small enough to read at a glance._
3. **Sections and fields: what is the page shape? Stick to Michael Nygard's classic Context / Decision / Consequences, or add more for metadata and relations?** — _`context` (prose — the forces and the problem), `decision` (blocks — the call, free to carry code or interface snippets), `consequences` (blocks — what becomes easier or harder), plus metadata: `date` (scalar ISO string), `deciders` (list), `scope` (scalar — the package/area, for filtering), `legacyId` (scalar — the original label), and a `relations` section holding the `supersededBy` ref. This is Nygard's template plus exactly the metadata the migration needs to be lossless._
4. **How is supersession represented and enforced? A free-text "superseded by ADR-x" note (rots, unverifiable), a typed graph `link` with a role (already in the engine, but lives outside the page's fields), or a `ref` field-kind (render-derived, integrity-checked)?** — _A `supersededBy` `ref` field on the record, set by a `supersede(targetId)` command, with the reverse "Supersedes" view derived from incoming refs. The `accepted → superseded` transition is precondition-gated on that ref being present AND pointing at a live `decision-record`; the engine's link integrity guarantees the target exists, so a superseded decision can never dangle. This is the load-bearing decision — it is what makes the decision graph trustworthy and is the direct fix for unversioned prose appendices._
5. **Migration of the existing ~27 ADRs: hand-author each, or write a parser? And do the DESIGN.md appendices stay after migration?** — _A one-off, re-runnable migration script (engine-as-library against a fresh "ADRs" workspace) parses each DESIGN.md appendix into a `decision-record`, preserving the legacy label, date, and scope, and setting `supersededBy` wherever the prose already says one decision replaces another. The appendices are deleted only in the same change that lands the disk snapshot (the linked projection feature), so there is never a window with two sources of truth and never one with none._
6. **One ADR workspace, or one per package? A workspace is one stream and one consistency aggregate.** — _One global "ADRs" workspace. A single aggregate is exactly what provides global identity and cross-package supersession — a `wiki-mcp` decision can supersede a `wiki` decision in one atomic graph. Package/area is captured by the `scope` field for filtering and grouping, not by splitting workspaces (which would re-introduce the per-file siloing this feature exists to remove)._
7. **Where does the git-resident `docs/adr/*.md` snapshot come from — is generating it part of this feature, or a separate concern?** — _No — generating the on-disk Markdown is the separate "Markdown projection to disk" feature (linked). This spec's only obligation toward it is constraint #2: render must be deterministic and stable-ordered, so that projection is churn-free. Keeping the two features separate keeps this one about the page type and the migration, and lets the projection serve the architecture and feature-spec workspaces too._
8. **Validated against the engine source — is setting a top-level `ref` field supported and integrity-checked today, or is an engine change required? And what is the final `supersede` mechanism (refining Q4)?** — _Validated: setting a top-level `ref` field is already fully supported — `setField` carries a whole `IField` incl `{kind:"ref",target}`, the reducer stores it, and ingestion enforces existence (`RefIntegrityError`; a page-ref resolves iff the target page exists). `architecture` already ships a page-ref field, so the path is proven. The ONLY gap is sugar: `kindFor` (command-bus) has no `ref` case, so a declarative `set: arg(id)` would store prose. This feature folds in that one small, GENERIC fix (refs first-class in `set:` for all models) — the schema-agnostic boundary holds. Refining Q4: supersession is a two-op atomic batch — `setSupersededBy(id)` (ingestion checks existence at set time) then `supersede()` (a `namesSuccessor` precondition reads the now-committed ref and checks it is a live `decision-record`). Two steps, because a precondition runs before its own command's ops and receives no args; `mutatePageBatch` makes it one commit._
