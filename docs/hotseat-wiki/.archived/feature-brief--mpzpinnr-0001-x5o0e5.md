# Feature: wiki-ui model inspection

**Status:** shipped

## Summary
wiki-ui renders each page as read-only Markdown, which hides the machinery that actually governs it: the page type's status FSM — the lifecycle states and the named, typed transitions between them — and, for the page in front of you, which of those transitions are legal right now. Today the only way to ask "what can I do to this page, and why can't I ship it yet?" is to call describeMutations by hand and read a flat list.

This feature adds a model-inspection view to wiki-ui that draws a page type's status FSM as a directed graph (states as nodes, transitions as labeled edges) and, for the currently-open page, highlights the transitions available from its current state. The current state is marked; legal transitions are emphasized; transitions that are FSM-reachable but blocked by an unmet precondition (e.g. `ship`, gated on a complete checklist and zero open questions) are shown distinctly, with their reason, so the "why not" is answerable at a glance.

Rendering goes through a real graph library rather than hand-drawn SVG, so layout, edge routing, and interaction (hover for transition metadata, select a state) come for free and the view scales to richer FSMs. Because wiki-ui already live-tails the workspace, the "available now" highlight updates the instant a page's status changes.

The point is to make the invisible governance of a structured page legible — turning its FSM from an implementation detail into a navigable map of where the page is and where it can go next. Note the two altitudes the view must keep distinct: the graph is a property of the page *type*; the highlight is a property of the specific page *instance*.

## Components affected
- wiki-ui — model-inspection view: the FSM graph component, current-state marker, and available/blocked-transition highlighting, plus where the view lives in the page UI
- wiki engine (wiki) — a public, serializable FSM-introspection surface: today the Guard (states/transitions/available/toMermaid) lives in wiki/src/core/guard.ts and is NOT exported
- wiki-mcp / wiki-server — optional FSM-metadata endpoint or MCP introspection tool, needed only if the UI must inspect models it did not statically bundle (runtime-loaded models)
- describeMutations (existing MCP tool / IPageView API) — reused as the live per-instance signal for which transitions are legal now and which are precondition-blocked

## Design constraints
1. Render the FSM with an established graph library — states as nodes, named transitions as directed, labeled edges. No hand-rolled SVG layout or ad-hoc positioning.
2. For the open page, the view must visually separate three classes: (a) the current state; (b) transitions legal right now; (c) transitions that are FSM-reachable from the current state but blocked by an unmet precondition — the last shown with its reason where one is available, so "why can't I ship?" is answerable directly from the graph.
3. The "available now" highlight must track live page status and re-render when it changes. Reuse wiki-ui's existing live-tail (the WorkspaceSession subscribe path); do not introduce polling.
4. Build on existing engine machinery rather than re-deriving FSM extraction: the Guard already exposes states(), transitions, available(status), and toMermaid(), and describeMutations already reports per-page command legality (including precondition gating). The rendered graph must be derived from the actual loaded page-type definition — the same one that drives mutations — so the diagram cannot drift from the behavior it documents.
5. Respect package boundaries: FSM-structure knowledge stays in the engine/registry (wiki) and schema (wiki-models); the UI consumes a machine-readable view of it and must NOT hardcode any page type's states or transitions.

## Resolved questions
1. **Graph library: Mermaid vs. an interactive layout+render library? The Guard already emits a Mermaid state diagram via toMermaid() (near-zero extraction work, but limited per-node/per-edge interactivity and dynamic highlighting), versus React Flow (@xyflow/react with a dagre/elk layout) or Cytoscape (full control over node/edge styling, hover, click, and live highlighting — at the cost of building the graph from states()/transitions and running layout). Leaning React Flow for the dynamic available-now highlighting, with Mermaid as a possible fast first cut.** — _React Flow (@xyflow/react). It gives first-class control over node/edge styling, hover, click, and the dynamic available/blocked highlighting the overlay needs, paired with a dagre or elk layout pass to position states. Mermaid/toMermaid() is not the rendering path — its per-edge dynamic styling is too limited — though it could remain a cheap static export if ever wanted._
2. **FSM data source — client-side or server endpoint? wiki-ui statically imports the model bundles and runs an in-browser engine, so it could derive the FSM from the page type's Guard directly with no backend change. But models are meant to be runtime-loaded by the server, and a UI that only knows statically-imported types cannot inspect a model it didn't bundle. Do we (a) derive client-side from the imported Guard now, or (b) add a server-side FSM-metadata endpoint / MCP introspection tool so the UI can inspect any loaded model? This is the load-bearing architectural choice; everything else follows from it.** — _The wiki engine itself describes the FSM for any loaded page type. The engine already holds every registered model (including runtime-loaded ones) in its registry, so it is the right and only place that can describe an arbitrary FSM. Both consumers then get it for free: the in-browser engine serves the client UI directly, and a thin server endpoint / MCP introspection tool exposes the same engine call for any other process. No FSM knowledge is duplicated in the UI or hardcoded per type. This is the load-bearing decision; Q3 (shape) and Q4 (instance overlay) follow from it._
3. **If the FSM is derived from the Guard (client- or server-side), what is the right public shape for exposing it? The Guard is internal to wiki today. Options: export a registry accessor (e.g. registry.pageGuard(type)), or add a serializable descriptor (e.g. fsmOf(type) -> { states, transitions: [{from, event, to, meta}], initial }) that both the UI and any endpoint can consume. Which package owns it, and should it be part of wiki's public surface or wiki/registry?** — _Follows from Q2: the wiki engine is the single thing that exposes this, as real typed objects rather than a string blob. The exact serialization is not load-bearing, but a typed descriptor is preferred — states plus transitions ({ from, event, to, meta }) and the initial state — so consumers can render it with React Flow, drive other tooling off it, or serialize it themselves however they like. The internal Guard graph becomes (or backs) this public typed surface instead of staying private to wiki/src/core/guard.ts._
4. **Source of "currently available": pure FSM or describeMutations? guard.available(status) yields the FSM-legal transitions, but describeMutations additionally factors in preconditions (e.g. ship is FSM-legal from review yet blocked until the checklist is complete and open questions are zero). Driving the highlight from describeMutations lets us render precondition-blocked transitions as "blocked — here's why" rather than misleadingly "available." Confirm describeMutations is the intended source for the instance overlay, with the Guard supplying the full type-level graph underneath.** — _Use describeMutations as the per-instance overlay; the engine's typed FSM descriptor (Q2/Q3) supplies the full type-level graph. The descriptor draws every state and edge (the map); describeMutations decides, for this page right now, which outgoing transitions are legal (paint green) vs. FSM-reachable but precondition-blocked (paint distinctly, with the reason — e.g. ship blocked until the checklist is complete and open questions are zero). describeMutations is the only source that runs preconditions, so it is required to honor constraint #2's third visual class. guard.available(status) alone would know only FSM-legality and would mislabel gated transitions like ship as available._
5. **Scope: page-level status FSM only, or element-level FSMs too? Several element types carry their own FSMs (a question is open → resolved; a plan step is todo ↔ done; a test case is planned → passed/failed). Is element-FSM visualization in scope for v1, or do we ship the page-status FSM first and defer element FSMs to a follow-up?** — _v1 covers the page-level status FSM only. Element-level FSMs (question open→resolved, plan step todo↔done, test case planned→passed/failed) are deferred to a follow-up. The typed FSM descriptor (Q3) should still be shaped so element FSMs can be added later without reworking it._
6. **Where does the view live: a side panel on the existing page view, a dedicated tab/route per page, or a modal/overlay? This affects how we present the per-type-vs-per-instance distinction (the graph is the type's; the highlight is this instance's) and how the live status binding is surfaced.** — _A button at the top of the page view toggles into the model-inspection view (page content ⇄ FSM graph). v1 is a view switch in the page header, not a side-by-side panel or a separate route — so the type-level graph and the live per-instance highlight are always one click from the page you're reading._
