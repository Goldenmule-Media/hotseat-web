# Feature: OAuth 2.1 login for MCP and CLI clients

**Status:** building

## Summary
Add an OAuth 2.1 authorization-server façade to wiki-server's existing auth gateway (GitHub OAuth + stateless HMAC-signed wsv1 bearer tokens) so MCP clients (Claude Code) and CLIs (wiki-mirror, wiki-mcp migrate-workspace) stop copying static tokens. The gateway gains RFC 8414/9728 metadata endpoints (/.well-known/oauth-authorization-server, /.well-known/oauth-protected-resource), stateless RFC 7591 dynamic client registration (the client_id is itself a signed blob), /auth/authorize (PKCE S256, piggybacking the existing GitHub dance by embedding the OAuth request in the signed wst1 state and minting a short-TTL signed wsc1 authorization code at the callback), and /auth/token (code + verifier → short-lived wsv1 access token + longer-lived signed wsr1 refresh token; refresh_token grant honored). The existing 401 www-authenticate header gains the RFC 9728 resource_metadata parameter so MCP clients auto-discover the login flow, and the embedded wiki-mcp HTTP transport's 401 gets the same via host-injected config. Clients get a loopback-redirect CLI login storing credentials at ~/.wiki/credentials.json (sibling of wiki-mirror.config.json), refreshed on demand through the already-shipped per-request IStreamConfig.headers function seam. Everything stays stateless — signed blobs reusing the existing sign/verify primitives, no session store, revocation remains signing-secret rotation — and the access token stays the wsv1 format so every existing verifying surface (gateway data plane, control listener, McpAuth) works unchanged.

## Components affected
- wiki-server/src/auth/tokens.ts — two new stateless token families beside wsv1/wst1: wsc1 authorization codes (signAuthCode/verifyAuthCode) and wsr1 refresh tokens (signRefreshToken/verifyRefreshToken), plus a PKCE S256 helper (sha256-base64url compare via the existing safeEqual)
- wiki-server/src/auth/oauth.ts (new) — pure OAuth 2.1 façade logic: RFC 8414/9728 metadata documents, RFC 7591 stateless client registration (signed client_id blob carrying redirect_uris), authorize-request validation, authorization_code and refresh_token grant handlers; gateway.ts stays thin routing
- wiki-server/src/auth/gateway.ts — route wiring: public GET /.well-known/* carved out of the deny-by-default data-plane proxy; POST /auth/register, GET /auth/authorize, POST /auth/token added before the bearer-session gate; the GitHub callback branches on an oauth-request payload in the verified state; unauthenticated() gains resource_metadata
- wiki-server/src/config.ts — new accessTokenTtlSeconds / refreshTokenTtlDays config (flags → env WIKI_SERVER_* → defaults), threaded into GatewayConfig from main.ts
- wiki-mcp/src/mcp/server.ts + wiki-server/src/main.ts — host-injected resourceMetadataUrl so the MCP transport's 401 advertises resource_metadata, and the MCP listener serves its own /.well-known/oauth-protected-resource pointing at the gateway as authorization server; McpAuth stays mechanism-agnostic
- Shared CLI OAuth client — new Node-only wiki/auth-client subpath (beside wiki/authoring · wiki/registry · wiki/testing): discovery fetch, dynamic registration, RFC 8252 loopback-redirect login, token exchange, ~/.wiki/credentials.json store (atomic temp+rename, mode 0600), and an oauthHeaders(serverUrl) factory returning an IStreamHeaders-compatible authorization function that refreshes via /auth/token when near expiry
- wiki-mirror — login subcommand plus credentials fallback in startMirror: static token (flags → WIKI_MIRROR_TOKEN → config file) still wins; otherwise authorization becomes a refreshing function instead of the current static Bearer string
- wiki-mcp/src/migrate-workspace.ts — streamConfig() gains the same credentials.json fallback behind --source-token/--dest-token/env
- Docs — .env.example, CLAUDE.md running-locally notes, a new ADR extending the GitHub-auth ADR, and this feature's spec pages

## Design constraints
1. Stateless philosophy is non-negotiable: authorization codes, refresh tokens, and client_ids are HMAC-signed blobs reusing the tokens.ts sign/verify primitives — no session store, no refresh-token ledger; revocation remains signing-secret rotation, which invalidates codes/refresh/access alike
2. The wsv1 access-token format must not change: the gateway data plane, control listener, and McpAuth all verify it identically today and must keep working untouched; OAuth only changes how clients OBTAIN wsv1 tokens and their TTL
3. No engine changes: the IStreamConfig.headers seam (header values may be per-request functions, evaluated in event-log.ts headerOpts()) is already sufficient for refresh; clients switch from static strings to functions
4. The gateway has NO cookie session — sessions are bearer-only — so /auth/authorize cannot check an existing login; it must always chain into the GitHub dance, carrying the pending OAuth request inside the signed wst1 state and minting the wsc1 code at /auth/github/callback (GitHub silently re-approves an already-authorized app, so UX is one redirect)
5. Metadata endpoints must be PUBLIC and live on the gateway, not the loopback-only control listener; /.well-known/* needs an explicit carve-out before the data plane's deny-by-default proxy
6. Existing routes are immutable: /auth/github, /auth/github/callback, /auth/config, /auth/me, /auth/workspaces/* and the token-in-URL-fragment browser flow keep working byte-for-byte; OAuth endpoints are additive in the /auth/* + /.well-known/* trees
7. PKCE S256 is required for all public clients; the code_challenge is embedded in the signed code blob and verified at /auth/token — no server-side challenge storage
8. wiki-server / wiki-mcp / wiki-mirror use relative imports with .js extensions (compiled Node packages); wiki-mirror must import no wiki-mcp/wiki-server internals; static-token paths (WIKI_MIRROR_TOKEN, --source-token/--dest-token, config-file token) remain supported and take precedence
9. Determinism in tests: injected nowSeconds clock (already the gateway pattern) and stubbed fetch (the githubStub pattern in auth-gateway.test.ts); no new heavy dependencies — node:crypto covers HMAC and PKCE SHA-256
10. Config follows the established flags → env → defaults precedence; allowedUsers (WIKI_SERVER_AUTH_USERS) must be re-checked at BOTH code mint and refresh grant so removing a user cuts them off at next refresh

## Resolved questions
1. **Single-use authorization codes: OAuth 2.1 requires codes be single-use, but a stateless signed blob cannot be marked consumed. Accept very short TTL (120s) + PKCE binding as the mitigation (consistent with the no-store philosophy), or add a tiny per-process in-memory seen-codes cache (fine single-node, lost on restart)?** — _Stay stateless: 120s code TTL + mandatory PKCE S256 binding is the mitigation; no seen-codes cache. A stolen code is useless without the verifier, and the window is two minutes. The ADR documents this as a deliberate, explicit deviation from OAuth 2.1 strictness, with secret rotation as the kill switch. Revisit only if the server goes multi-tenant._
2. **Refresh-token rotation semantics: OAuth 2.1 expects rotation with reuse detection for public clients; statelessly we can rotate (new wsr1 per refresh, exp capped at the original) but CANNOT detect reuse of the old one until expiry. Is capped rotation without reuse detection acceptable, or should refresh tokens be fixed (no rotation)?** — _Capped rotation without reuse detection: each refresh mints a new wsr1 whose exp never exceeds the ORIGINAL grant's exp, so a leaked refresh token cannot extend its own life. Reuse detection genuinely requires a store; pretending otherwise would imply a guarantee we don't have. The ADR states the gap explicitly._
3. **Home of the shared CLI OAuth client: a new Node-only wiki/auth-client subpath export (engine package gains node:fs/node:http in an isolated subpath — wiki-ui simply never imports it) vs. duplicating a small module in wiki-mirror and wiki-mcp (no boundary risk, but two copies)?** — _New Node-only wiki/auth-client subpath export. One implementation beats two drifting copies, and the precedent exists: wiki/testing is already a Node-leaning subpath. The engine core stays transport-free — the subpath is opt-in, never imported by wiki-ui or the browser path, and bundlers only see it if asked._
4. **CLI flow: ship loopback-redirect (RFC 8252) only, or also device-code (RFC 8628) for headless/SSH machines where no local browser can open the authorize URL? Device-code adds two endpoints (/auth/device, polling on /auth/token) and a verification-URI page.** — _Loopback-redirect (RFC 8252) only for v1 — both current consumers (wiki-mirror login, migrate-workspace) run on machines with a browser. Device-code (RFC 8628) is deferred; the oauth.ts façade is shaped so adding /auth/device later is additive._
5. **TTL defaults and the wiki-ui session: keep the interactive GitHub-callback fragment token at sessionTtlDays (30d wsv1, today's behavior) while OAuth clients get 1h access + 30d refresh — or move wiki-ui onto the short-access+refresh model too (more work in wiki-ui/lib/auth.ts, better hygiene)?** — _Keep the wiki-ui interactive flow exactly as-is (30d wsv1 in the URL fragment); OAuth clients get accessTokenTtlSeconds=3600 access + 30d refresh. Moving wiki-ui onto short-access+refresh is better hygiene but a separate feature — this one is about MCP/CLI credentials, and scope stays focused._
6. **Token scope/resource: keep tokens global-per-user with per-request ledger checks (the existing per-surface enforcement; simplest), or honor RFC 8707 resource indicators to mint audience-scoped tokens (stream vs MCP)?** — _Global-per-user tokens. The existing per-surface enforcement (every surface re-checks the ledger per request; identity is never trusted across surfaces) already provides the isolation RFC 8707 audiences would. The resource parameter is accepted and carried in the code blob for forward-compatibility but does not scope the minted token._
