# Feature: Shared engine in a SharedWorker (one engine + PGlite across all tabs)

**Status:** planning

## Summary
Today wiki-ui runs the full `wiki` engine — and its own PGlite-on-IndexedDB — once **per tab**. `lib/engine.ts` creates one `IWiki` per tab via `createWiki({ stream, pageTypes, search:{ db } })`, and `lib/search-db.ts` opens `new PGlite("idb://wiki-ui-search")` per tab as the engine's full-text-search read model. So N tabs means N engines, N Durable-Stream tails, N folds, N reachability probes — and, the real hazard, N PGlite instances all writing the SAME IndexedDB store. PGlite's `idb://` FS is single-connection/single-writer (it loads the data dir into memory and flushes changed files back after each query), so concurrent tabs are diverging private snapshots racing to last-flush-wins. The blast radius is bounded to *search* (the index is a derived projection with a per-workspace `applied_version` cursor that self-heals on the next clean fold; page content comes from the stream, and the UI never queries PGlite directly) — but cross-tab search flakiness is a real, latent bug.

This feature moves the engine **and** PGlite into a single **SharedWorker**. Tabs become thin RPC clients over the worker's `MessagePort` (via Comlink); the worker owns the one engine, the one PGlite (one writer), the one stream tail, one fold per workspace, the one reachability probe, and the one-time search priming. The React UI store (`useSyncExternalStore` in `lib/live.tsx`) stays in the tab, as does all pure CPU work — `marked` markdown→HTML, the Dagre/React-Flow FSM layout, and schema-form generation — which operate on plain, structured-clone-safe return data. The split is clean: anything that touches the live engine instance moves into the worker; anything that is pure computation over its return values stays in the tab.

**Scope decision (deliberate): SharedWorker-only, no fallback.** Chromium-on-Android and Safari < 16 are explicitly out of scope. A **module** SharedWorker (`{type:"module"}`) is required because the worker imports the engine as ESM, which narrows support to Chrome/Edge, Firefox 114+, and Safari 16+. We accept that gap in exchange for one clean code path — no Web-Locks leader-tab election, no dedicated-worker-per-tab variant to maintain. Where SharedWorker is absent we feature-detect and fail loudly with an unsupported-browser message rather than degrade.

Feasibility is settled by the transport: the Durable-Streams client tails via `fetch` + `response.body.getReader()` (SSE-over-fetch), **not** the `EventSource` API — which matters because `EventSource` is absent in `SharedWorkerGlobalScope`, whereas `fetch`, `ReadableStream`, `TextDecoder`, IndexedDB, and WASM are all present. So the engine and PGlite run inside a SharedWorker unmodified; `wiki`/`wiki-models` are not touched. The entire cost of this feature is the tab↔worker boundary — the RPC seam, the subscription fan-out, and the error protocol — not the engine.

## Components affected
- NEW `lib/wiki-host.worker.ts` — the SharedWorker entry. Builds PGlite + Kysely (`PGliteDialect`) + the `pageTypes` bundle and calls `createWiki` in worker scope; owns the single engine, the single stream tail, the reachability probe (moved off the tab), and the one-time search priming. `onconnect` → `Comlink.expose(api, port)` per connecting tab. Exposes the collapsed RPC set, a per-port commit broadcast, the error transfer handler, and a heartbeat reaper that drops a tab's subscriptions when its port goes silent.
- NEW `lib/host-client.ts` — the tab side. Feature-detects (`typeof SharedWorker !== "undefined"`) and, by decision, throws a clear unsupported-browser error when absent (no fallback). Constructs the worker with a static `new SharedWorker(new URL("./wiki-host.worker.ts", import.meta.url), { type:"module" })` literal (a hoisted URL defeats bundler static analysis), `Comlink.wrap`s the port, performs the handshake, and caches the immutable `FsmDescriptor` set the worker pushes so `fsmOf` stays synchronous in render. Exposes the async host the React stores consume.
- `lib/engine.ts` — `getWiki()` stops calling `createWiki` in the tab; `createWiki` moves into the worker. `pageTypes` (closures) and `search.db` (a live Kysely/PGlite handle) are non-cloneable and *must* be constructed worker-side — they never cross the port. The tab-side replacement is an async `getHost()` returning the Comlink proxy; the current synchronous `getWiki(): IWiki | null` contract becomes a three-state "connecting" handle, preserving the existing null-on-SSR guard.
- `lib/search-db.ts`, `lib/pglite-dialect.ts`, `lib/models.ts` (the `pageTypes`) — relocate **into the worker**, code essentially unchanged. One PGlite owner ends both the multi-idb write hazard and the per-tab `migrateSearchToLatest` race. The `db: Kysely<…>` injection seam is exactly why this is clean: a Kysely handle is non-serializable, so whoever runs `createWiki` must own the DB in its own realm — which is now the worker.
- `lib/live.tsx` — **stays in the tab as the UI store**, internals rewired. The `WorkspaceSession` snapshot cache + `useSyncExternalStore` seam survive (the snapshot is already fed async). `h.tree()` → RPC `tree(ws)`; `h.subscribe(cb)` → subscribe to the worker's per-port broadcast (+ unsubscribe-on-port-close + heartbeat); `usePage`/`usePageMutations` collapse the `h.page(id).describeMutations()` two-hop into one RPC; `classify()` branches on a serialized error `code`, not `instanceof`.
- `lib/search.ts` — the command-palette UI state stays per-tab, but the `ensureIndexed` priming (which folds every workspace once so search spans all of them) moves **into the worker**: it is a property of the one shared index, not of each tab, so a freshly-opened tab must not re-fold everything. `wiki.search(...)` and `listWorkspaces()` become plain RPCs.
- Error protocol (cross-cutting) — at the worker boundary, every engine throw is caught and serialized to a plain DTO `{ code, message, types?, issues? }`, read off the live instance worker-side where `instanceof`/own-props are intact. Tab-side, `classify()` already duck-types `.code`/`.types`, so the DTO flows straight through; optionally re-`new` the matching `WikiError` subclass from `code` to restore `instanceof` for any other consumer. Implemented as a Comlink custom `transferHandler` registered on both sides.
- Tab-side pure modules stay put — `components/PageView.tsx` + `lib/markdown.ts` (`marked` → HTML + link rewrite), `components/FsmGraph.tsx` + `lib/fsm-graph.ts` + `lib/floating-edge.ts` (Dagre layout + React Flow), and `lib/schema-form.ts` + `components/TransitionForm.tsx` (schema → form → coerced args). All operate on cloned plain data (`ITreeNode`, `IMutationDescriptor[]`, the `toMarkdown` string, `FsmDescriptor`). `fsmOf` stays synchronous over the descriptor cache pushed at handshake — it must NOT become an RPC, since it is called in render.

## Design constraints
1. SharedWorker-only, no fallback. Explicitly out of scope: Chromium-on-Android and Safari < 16. A module SharedWorker (`{type:"module"}`, needed because the worker imports the engine as ESM) requires Chrome/Edge, Firefox 114+, Safari 16+. Feature-detect and fail loudly where absent; do NOT build a Web-Locks leader-tab or per-tab-worker fallback — the single clean path is the point.
2. One engine, one connection, one writer. Exactly one `createWiki`, one Durable-Stream tail, one fold per workspace, and one PGlite instance against `idb://wiki-ui-search`, all owned by the worker. This removes the multi-tab PGlite hazard by construction rather than by coordination.
3. The engine is unchanged. `wiki` and `wiki-models` are not modified; the worker imports their public surface exactly as the tab does today (via Next `transpilePackages`). All new code lives in `wiki-ui`. The `wiki` package already proves it is transport-free and worker-safe.
4. Cleanest possible boundary. Collapse the `IPageView` facade away — expose one coarse `describeMutations(ws, page)` RPC instead of a `handle → page → method` chain of round-trips. Push all immutable `FsmDescriptor`s to the tab once at handshake so `fsmOf` stays synchronous in render. Keep the RPC surface small and coarse-grained; never proxy a chatty fine-grained object graph across the port.
5. Errors cross as data, not exceptions. The engine's typed errors (`WikiError.code`, `UnknownPageTypeError.types`) lose their class identity AND their custom own-properties under structured clone — a naive boundary delivers `code === undefined` and mis-buckets every engine error as a connection failure, destroying the connection-vs-schema distinction `classify()` is built on. The boundary MUST marshal `{ code, message, types? }` explicitly.
6. Live updates stay push, never poll. The worker owns the single tail and broadcasts each commit — carrying the affected workspace + pageId(s), and ideally the freshly-folded tree it already holds — to subscribed ports, so a tab re-pulls only what changed rather than re-reading `tree()` + `toMarkdown()` + `describeMutations()` on every event (cheap in-process, an RPC storm across the boundary). `useSyncExternalStore` remains the React seam.
7. Per-tab lifecycle is the worker's responsibility. SharedWorker has no reliable port-closed event, so the worker reaps a tab's subscriptions via a heartbeat/ping; the engine's `teardown()` (which clears all subscribers) runs only when the LAST tab disconnects. Today `live.tsx` never unsubscribes (sessions live for the tab) — harmless per-tab, a fan-out-to-dead-ports leak with a shared host.
8. SSR discipline preserved. `SharedWorker` and `PGlite` are client-only; instantiate lazily behind `typeof window` / `typeof SharedWorker` from a `"use client"` boundary, never at module load. The current `getWiki()` null-on-server contract is preserved as the handle's "connecting" state so client components still pre-render cleanly.

## Resolved questions
1. **SharedWorker, a Web-Locks leader-tab, or just share the DB via PGlite's `PGliteWorker`?** — _SharedWorker. It is the cleanest of the three: the OS hosts one engine that outlives any individual tab, with no leader election and no re-fold on leader handoff. A Web-Locks leader-tab gives the same one-engine/one-DB end-state but adds election + handoff complexity and runs the engine on a real tab's main thread; `PGliteWorker`-only shares just the DB and leaves N engines each folding the stream. Both are rejected in favour of the single clean path._
2. **What about browsers without SharedWorker (Chromium-on-Android, Safari < 16)?** — _Out of scope by decision. We accept the support gap (Chromium-on-Android, Safari < 16) to keep one clean code path rather than maintain a fallback. The client feature-detects `SharedWorker` (and module-worker support) and shows an unsupported-browser message where absent. No Web-Locks leader-tab, no per-tab dedicated worker._
3. **Does the engine's live tail actually work inside a SharedWorker (no `EventSource` in worker scope)?** — _Yes. The Durable-Streams client tails over `fetch` + `response.body.getReader()` (it parses SSE in JS over the `ReadableStream`); it never uses the `EventSource` API — which is the one relevant transport that is absent in `SharedWorkerGlobalScope`. `fetch`, `ReadableStream`, `TextDecoder`, IndexedDB, and WASM are all available in a worker, so the engine + PGlite run there with no changes to `wiki`._
4. **How do the engine's typed errors survive the tab↔worker boundary?** — _Marshalled as a plain DTO `{ code, message, types?, issues? }` at the worker edge (read off the live error instance, where the prototype and own-props are intact), because structured clone of an `Error` keeps only `name`/`message`/`stack` and drops the subclass identity plus the custom `.code`/`.types` fields. Tab-side, `classify()` already duck-types `.code`/`.types`, so the DTO flows straight through with no class identity needed; a Comlink custom `transferHandler` does this symmetrically on both sides. This is the single sharpest edge — without it every engine error mis-reports as a connection failure._
5. **Does moving the engine into the worker move PGlite cleanly, or does the UI also need direct DB access?** — _Cleanly. PGlite reaches the engine only through the `search:{ db }` injection seam and has exactly one consumer — the engine itself; the UI never queries the DB directly (all search goes through `wiki.search(...)`, page content comes from the stream). So relocating `createWiki` into the worker relocates PGlite with it, and `pglite-dialect.ts` + `migrateSearchToLatest` move unchanged. A live Kysely handle is non-serializable anyway, which is precisely why the DB must live in the same realm as the engine._
